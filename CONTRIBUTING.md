# Contributing to okf-site

Thank you for your interest in contributing. Contributions are very welcome,
whether you are fixing a typo in the copy, improving accessibility, tightening
the markup, or adding to the AI-discoverability files.

`okf-site` is the landing page for [okf](https://github.com/okfcli/okf), the
Open Knowledge Format CLI. It is a single [Cloudflare Worker](https://developers.cloudflare.com/workers/)
that serves one inline HTML page, deployed to **useokf.com**. There is no
framework and no build step. The only client-side JavaScript is the DataFast
analytics snippet.

## Code of conduct

This project is governed by our [Code of Conduct](CODE_OF_CONDUCT.md). By
participating, you are expected to uphold it. Report unacceptable behavior to
accounts+conduct@akeemjenkins.com.

## AI-assisted contributions are welcome

Contributions written with the help of AI tools (Claude Code, Copilot, Cursor,
and others) are welcome. A good pull request is a good pull request regardless
of how it was produced. We ask a few things when AI helped write your change:

- **You are the author and you are accountable.** Read, understand, and stand
  behind every line you submit. If a reviewer asks why a line exists, you should
  be able to answer.
- **Run it locally before you open the PR.** At minimum, preview the page with
  `npm run dev` and confirm it bundles with the dry-run below. AI-generated code
  that was never run wastes reviewer time.
- **Do not paste in code, copy, or assets you do not have the right to
  contribute.** Only submit work you wrote or are licensed to contribute under
  Apache 2.0.
- **Keep the diff reviewable and focused.** Do not submit a sprawling
  machine-generated rewrite with unrelated churn.
- **Disclose substantial AI authorship** in the PR description. A short note is
  appreciated and builds trust. It is not required for small changes.

The same quality bar applies to all contributions. AI assistance does not lower
it and does not raise it.

## Getting started

### Prerequisites

- **Node.js** (a recent LTS release) and **npm**.
- A [Cloudflare account](https://dash.cloudflare.com/) only if you intend to
  deploy. You do not need one to develop and preview locally.

### Set up the repo

```bash
git clone https://github.com/okfcli/okf-site.git
cd okf-site
npm install
npm run dev      # local preview at http://localhost:8787
```

## Project layout

- `src/index.js`: the entire site. A `handler` routes `/robots.txt`,
  `/llms.txt`, `/llms-full.txt`, `/sitemap.xml`, and otherwise serves the `HTML`
  template-literal page. The handler is wrapped for bot tracking and exported as
  the Worker `fetch`.
- `wrangler.toml`: Worker config. `routes` binds the `useokf.com` custom domain.
- `package.json`: deps `@datafast/ai-crawl` (bot tracking); devDeps `wrangler`.
  Scripts `dev` and `deploy`.
- `CLAUDE.md`: how the DataFast analytics, goals, and bot tracking are wired,
  plus deploy notes and copy style. Read this before touching analytics or the
  discoverability files.

## Development workflow

1. **Open an issue first for anything non-trivial.** Typos and small copy or
   markup fixes can go straight to a PR. For redesigns, new sections, or
   analytics changes, open an issue so we can align first.
2. **Branch from `main`** with a short descriptive name.
3. **Make your change** in `src/index.js` (the whole page lives there).
4. **Preview and verify** (see below).
5. **Open a pull request** against `main` with a clear description.

## Editing the page

- The whole page is one `HTML` template literal. It interpolates
  `${DATAFAST_WEBSITE_ID}`, so that constant must stay declared **above**
  `HTML`.
- Terminal examples use a bare `$` (for example `<span class="cmd">$</span>`),
  which is safe. Do not introduce `${...}` into copy unless you intend template
  interpolation, or the Worker will fail to build.
- Keep it dependency-free and build-step-free. The value of this site is that it
  is one small Worker. Do not add a framework or a bundler.
- **Accessibility matters.** Preserve semantic HTML, alt text, sufficient color
  contrast, and keyboard navigability.

### AI discoverability files

The site exposes the files LLMs and AI crawlers look for, all generated by the
Worker from constants in `src/index.js`: `/llms.txt`, `/llms-full.txt`,
`/sitemap.xml`, `/robots.txt`, plus embedded JSON-LD. When you change the
landing-page copy or the command set, keep `LLMS_TXT`, `LLMS_FULL_TXT`, and the
JSON-LD in sync. See `CLAUDE.md` for details.

### Analytics and goals

DataFast analytics and goals are documented in `CLAUDE.md`. Goals are tracked
with a `data-fast-goal="<name>"` attribute. If you add a call to action, wire an
appropriate goal and note it in your PR.

## Copy style (important)

The owner does **not** use em-dashes. Never introduce them into any copy, in the
page or the docs. Use commas, colons, or periods instead. Hyphens in compound
words (`agentic-first`, `cosign-signed`) are fine.

## Before you submit

Confirm the Worker still bundles without shipping:

```bash
npx wrangler deploy --dry-run --outdir /tmp/wgbuild
```

Also load the page locally with `npm run dev` and click through it, including
any links or goals you touched.

## Deploying

Deploys are handled by the maintainer. `npm run deploy` runs `wrangler deploy`
and pushes live to useokf.com, which requires Cloudflare auth. You do not need
to deploy to contribute. Just get your change green locally and open the PR.

## Pull request guidelines

- **Title:** a concise summary of the change.
- **Description:** what changed and why. Link the issue with `Closes #123` where
  applicable. Note any copy, analytics, or discoverability-file changes.
- **Scope:** one coherent change per PR.
- **Verify:** confirm the dry-run bundle passes and the page looks right locally.
- Be responsive to review feedback. We aim to be prompt and constructive in
  return.

## Reporting bugs

Open an issue describing what is wrong (broken link, layout issue, incorrect
copy, an AI-discoverability file out of sync), how to reproduce it, and a
screenshot if it is visual.

## Security issues

Please do not open a public issue for security vulnerabilities. Email
accounts+conduct@akeemjenkins.com with details and we will coordinate privately.
